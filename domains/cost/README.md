> Phase 1.5 refactors and Phase 2 DDD relocation.

## Purpose

The Cost domain owns portfolio spend truth: Azure usage costs, budgets,
reservation utilisation, anomaly review, and chargeback/showback reporting. It
turns raw billing feeds into tenant-scoped facts that operators can query,
compare, and export without crossing tenant authorization boundaries.

This domain answers:

- What did each tenant/subscription/resource/service cost over a period?
- Which spend changed materially compared with the prior period?
- Which cost anomalies need review and who acknowledged them?
- Are configured budgets and thresholds current with Azure Cost Management?
- What costs should be attributed back to a tenant, resource group, or service?
- Are Azure reservations under-utilised enough to need action?

## Invariants

1. **Tenant authorization is mandatory at every HTTP entry point.** Cost reads,
   anomaly acknowledgement, budget CRUD, reservation summaries, and chargeback
   exports must be constrained to `TenantAuthorization.accessible_tenant_ids`.
2. **Raw spend facts are tenant-scoped.** Cost facts without a tenant boundary
   are import errors, not platform facts.
3. **Money is never PII by itself, but attribution can be sensitive.** Resource
   names, tags, invoice descriptions, and acknowledgement users can identify a
   person, brand, vendor, or project.
4. **Budget writes are local-first but Azure-backed.** Budget CRUD must keep
   local records and Azure Cost Management state consistent or return a visible
   failure.
5. **Anomaly acknowledgement is auditable.** Acknowledgements record actor and
   timestamp; bulk acknowledgement must not bypass tenant authorization.
6. **Forecasts and anomaly scores are advisory.** They must not mutate budgets,
   resolve alerts, or trigger financial actions without an explicit domain
   command.
7. **External billing APIs are adapters.** Azure Cost Management, Azure
   Consumption Reservations, future Pax8, and future SaaS invoice ingestion are
   outbound interface implementations, not domain logic.
8. **Cross-domain access goes through interfaces.** Cost may reference tenant
   identity/metadata through a shared tenant read model or interface; it must not
   import identity, compliance, resource, lifecycle, or BI domain internals.

## Inbound Interface Contracts

### HTTP API

Owned route prefix: `/api/v1/costs`.

Current commands/queries:

- `GET /summary` — returns portfolio or tenant-filtered `CostSummary`.
- `GET /by-tenant` — returns tenant-scoped cost totals.
- `GET /trends` — returns daily cost trend points.
- `GET /forecast` — returns advisory forecast for a requested day count.
- `GET /anomalies` — returns anomaly list, tenant-filtered by route guard.
- `GET /anomalies/trends`, `/anomalies/by-service`, `/anomalies/top` — anomaly
  analytics.
- `POST /anomalies/{id}/acknowledge` and `POST /anomalies/bulk-acknowledge` —
  mutation commands; must validate tenant access before acknowledgement.
- `GET /reservations` — reservation-utilisation query for one accessible tenant.
- `GET /chargeback/{tenant_id}` and `GET /chargeback` — JSON/CSV showback
  exports.

Budget routes are currently under `/api/v1/budgets` but remain Cost-domain HTTP
surface:

- Budget CRUD: list/create/get/update/delete.
- Budget alert list and acknowledgement.
- Budget summaries and sync triggers.

### Sync commands

The application scheduler or sync API may request:

- `sync_costs()` for Azure usage-cost ingestion.
- `sync_budgets_for_tenant()` or `sync_all_tenant_budgets()` for Azure budget
  ingestion and threshold checks.
- Cost historical backfill through the generic backfill engine with job type
  `costs`.

### Data queries

Cost accepts read requests from dashboard/export/UI composition layers. Those
callers receive DTOs or exported files; they do not receive ORM sessions,
unfiltered rows, or cross-tenant raw billing dumps.

## Outbound Interface Contracts

Cost is allowed to call these interfaces/adapters:

| Interface | Current concrete implementation | Contract |
|---|---|---|
| Tenant read model | `Tenant` ORM via `Session` | Read tenant ID, name, active state, subscription IDs, and billing account metadata. No tenant mutation from Cost. |
| Authorization | `TenantAuthorization` | Filter requested tenant IDs and enforce at least one accessible tenant. |
| Azure Cost Management usage | `app/core/sync/costs.py` REST path and Azure SDK clients | Pull usage/cost rows by subscription and date range; return normalized cost facts. |
| Azure Budget API | `BudgetService._fetch_budgets_from_azure()` and create/update/delete helpers | Mirror Azure budgets into local budget records and push budget CRUD changes. |
| Azure Consumption Reservations | `ReservationService` | Read reservation utilisation for a billing account; no reservation purchase/exchange actions. |
| Monitoring | `MonitoringService`/`SyncJobLog` | Emit sync status and alert facts; do not own monitoring persistence. |
| Cache | `cached`, `invalidate_on_sync_completion` | Cache read queries; invalidate after successful cost/budget sync. |
| Notification | Budget alert notification models | Record notification preferences; send operations should remain in shared notification adapter. |

## Explicit Non-Goals

- Cost does not own tenant onboarding, tenant credential resolution, or Entra
  identity policy.
- Cost does not own resource inventory beyond cost dimensions copied from billing
  data. Resource truth belongs to the Resources domain.
- Cost does not own compliance evidence, even if cost anomalies are shown on an
  executive/compliance dashboard.
- Cost does not own lifecycle playbooks, but lifecycle may call Cost interfaces
  to estimate acquisition/run-rate impact.
- Cost does not own BI warehouse semantics; BI may ingest or query approved cost
  read models through a published interface.

## Phase 1.5 Refactor Guidance

1. Split `app/api/services/budget_service.py` along the boundaries already in
   the file: CRUD, alerting/thresholds, Azure sync adapter, and DTO mapping.
2. Split cost sync adapters from normalization/persistence so Azure API details
   do not leak into domain rules.
3. Keep route modules thin: authz, validation, service invocation, response.
4. Do not introduce lateral imports from future `domains/identity`,
   `domains/compliance`, `domains/resources`, `domains/lifecycle`, or
   `domains/bi_bridge`.
5. Prefer interfaces over concrete Azure SDK calls in domain services. Boring,
   testable seams beat heroic SDK spelunking every time.
