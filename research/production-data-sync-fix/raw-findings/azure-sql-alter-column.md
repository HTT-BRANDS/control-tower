## Changing Column Size — Official Documentation

> You can change the length, precision, or scale of a column by specifying a new size for
> the column data type. Use the `ALTER COLUMN` clause. If data exists in the column, the
> new size **can't be smaller than the maximum size of the data**. Also, you can't define the
> column in an index, unless the column is a **varchar, nvarchar, or varbinary** data type
> and the index isn't the result of a `PRIMARY KEY` constraint.

### Key Takeaways

1. **Widening is always safe** — increasing NVARCHAR length never fails
2. **Narrowing is conditional** — new size must accommodate existing data
3. **NVARCHAR in indexes is fine** — can be resized without index issues (unless PRIMARY KEY)
4. **Statistics may need dropping** — if auto-created stats exist on the column, `DROP STATISTICS` may be needed first

## Locks and ALTER TABLE — Official Documentation

> Changes you specify in `ALTER TABLE` implement immediately. If the changes require
> modifications of the rows in the table, `ALTER TABLE` updates the rows. `ALTER TABLE`
> acquires a **schema modify (Sch-M) lock** on the table to make sure that no other
> connections reference even the metadata for the table during the change, except online
> index operations that require a short Sch-M lock at the end.

> The modifications made to the table are logged and fully recoverable. Changes that affect
> all the rows in large tables, such as dropping a column or, on some editions of SQL Server,
> adding a `NOT NULL` column with a default value, can take a long time to complete and
> generate many log records. Run these `ALTER TABLE` statements with the same care as any
> `INSERT`, `UPDATE`, or `DELETE` statement that affects many rows.

### What This Means for NVARCHAR Widening

| Operation Type | Lock Duration | Data Modification | Impact |
|---------------|---------------|-------------------|--------|
| **Widen NVARCHAR** (our case) | **Milliseconds** | **None** (metadata-only) | **Negligible** |
| Add NOT NULL column w/ default | Long | Yes (all rows) | Significant |
| Drop column | Long | Yes (all rows) | Significant |
| Change data type | Long | Yes (all rows) | Significant |

**Widening NVARCHAR(500) → NVARCHAR(1000) is metadata-only**: SQL Server stores NVARCHAR as
variable-length data. The column metadata records the maximum allowed length. Increasing this
maximum only changes the metadata, not the actual stored data.

## Azure SQL S0 Tier Specifics

From Azure SQL DTU documentation:

> The Basic, S0, S1 and S2 service objectives provide less than one vCore (CPU). For
> CPU-intensive workloads, a service objective of S3 or greater is recommended.

> In the Basic, S0, and S1 service objectives, database files are stored in Azure Standard
> Storage, which uses **hard disk drive (HDD)**-based storage media. These service objectives
> are best suited for development, testing, and other infrequently accessed workloads that
> are less sensitive to performance variability.

### S0 Tier Resources

| Metric | S0 Value |
|--------|----------|
| DTUs | 10 |
| Storage | 250 GB |
| Max concurrent workers | 60 |
| Max concurrent sessions | 600 |
| Tempdb size | 32 GB |

### Impact on ALTER COLUMN

- S0 has **10 DTUs** — plenty for a metadata-only ALTER COLUMN
- HDD storage is irrelevant — metadata changes happen in buffer pool
- **Max concurrent workers: 60** — Sch-M lock may briefly queue a few workers
- For our table sizes (hundreds to low thousands of rows), zero practical impact

## Production Safety Checklist

### Before Running Migration

```sql
-- Check existing max data lengths
SELECT
    MAX(LEN(policy_definition_id)) as max_def_id_len,
    MAX(LEN(policy_name)) as max_name_len,
    MAX(LEN(policy_category)) as max_category_len
FROM policy_states;

-- Check for statistics on columns being altered
SELECT s.name AS statistics_name,
       c.name AS column_name,
       sc.stats_column_id
FROM sys.stats AS s
INNER JOIN sys.stats_columns AS sc ON s.object_id = sc.object_id AND s.stats_id = sc.stats_id
INNER JOIN sys.columns AS c ON sc.object_id = c.object_id AND c.column_id = sc.column_id
WHERE s.object_id = OBJECT_ID('policy_states');
```

### Expected Alembic Output

```sql
-- Generated by Alembic (metadata-only operations, sub-second each)
ALTER TABLE policy_states ALTER COLUMN policy_definition_id NVARCHAR(1000) NOT NULL;
ALTER TABLE policy_states ALTER COLUMN policy_name NVARCHAR(500) NOT NULL;
ALTER TABLE policy_states ALTER COLUMN policy_category NVARCHAR(1000);
```

### Rollback Strategy

If issues occur (extremely unlikely for metadata-only changes):
1. Alembic downgrade: `alembic downgrade 008`
2. **Warning**: Downgrade will fail if any data in the widened columns exceeds the old column size
3. If data exceeds old limits, truncate first:
   ```sql
   UPDATE policy_states SET policy_definition_id = LEFT(policy_definition_id, 500)
   WHERE LEN(policy_definition_id) > 500;
   ```

## Conclusion

**ALTER COLUMN to widen NVARCHAR on Azure SQL S0 is completely safe for production:**
- Metadata-only operation (no data rewrite)
- Sub-millisecond Sch-M lock duration
- No special S0 tier restrictions
- Compatible with existing indexes
- Standard Alembic `alter_column()` syntax
- Can run during normal application operation
